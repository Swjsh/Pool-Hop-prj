## 2026-07-02 (SensorLight false-alarm closure) — The "light visibility toggle never works" issue (open since the prior session, marked low-priority/unresolved in LESSONS) was NEVER a bug — it's the exact same "decay races round-trip latency" measurement trap as the heat-bump and VFX entries below, just not recognized as that family at the time

**Root cause, found by reading the actual graph instead of re-running the same style of test:** `BP_SensorLight.HandleSensorOverlap` calls `Rendering|SetVisibility(SensorSpotLight, true)` on trip, immediately followed by `Utilities|Time|SetTimerbyFunctionName(self, "TurnOffLight", LightOnDuration, ...)` — a **4-second** (`LightOnDuration` CDO default) delayed call to a function (`TurnOffLight`) that itself calls `Rendering|SetVisibility(SensorSpotLight, false)`. The prior investigation's "ruled out" list (stale/duplicate event nodes, collision-profile mismatch, stray `EventTick`/`EventBeginPlay` logic, `Mobility`) never considered a **timer-scheduled delayed function call** as a source of an auto-off — reasonable, since it isn't an event node at all, but it's functionally identical to one. Any test methodology that takes more than ~4 seconds of wall-clock between triggering the overlap and reading `bVisible` (very easy to do across several "exhaustive verification" MCP round-trips, each of which has real latency) would reliably observe `bVisible:false` and conclude the toggle "never works" — exactly what happened.

**Confirmed by testing with a MINIMAL round-trip** instead of a thorough one: read baseline `bVisible` (`false`) → one `set_actor_transform` teleporting the player into the `TriggerBox` → **immediately, next call, zero other calls in between** → `get_properties(SensorSpotLight, ["bVisible"])` → **`true`**. A few calls later (comfortably past 4s of real elapsed round-trip time), read again → back to `false` — the full on→off cycle, exactly as designed. **The light, the trigger, the timer, and the visibility toggle are all completely correct and were probably correct the entire time** — this was purely an artifact of how thoroughly (i.e., slowly) the previous session tested it. **Generalizes, and sharpens the standing "decay races round-trip latency" lesson (see the NS_NoiseRing/heat-bump entries below): before spending a long investigation on "X is set but doesn't stick", grep the surrounding graph for ANY delayed-execution node (`SetTimer*`, `Delay`, a dispatcher scheduled from `Tick`) that targets the same property — a thorough, multi-step live test is not automatically more trustworthy than a fast one; here, the SLOW test was the one that was wrong.**

## 2026-07-02 (vision-cone decal wiring) — Getting an `AIController`'s possessed pawn is `Pawn|GetControlledPawn`, not `Class|Controller|GetPawn` / `Variables|Default|GetPawn` (both genuinely don't exist); and where you splice new per-tick logic into an existing function matters when it reads a value that function itself just computed

**The node:** needed the AIController's own pawn from inside `BP_WatcherController.TickBrain` (to cast it to `BP_WatcherCharacter` and push live `AlertState`/`DetectionMeter` into its new vision-cone decal material). Two intuitive guesses both failed outright with `"...does not exist"`: `Class|Controller|GetPawn` and `Variables|Default|GetPawn`. **The real type_id is `Pawn|GetControlledPawn`** — found via `find_node_types(type_id_filter: "ControlledPawn")` after the obvious guesses failed. It's a **pure** node (no exec pins) with a `self` input pin typed `Controller Object Reference` and one output `ReturnValue` typed `Pawn Object Reference` — wire it straight into a cast, no exec plumbing needed. (A stray, unrelated error message earlier in the session had referenced `CallFunc_K2_GetPawn_ReturnValue`, which looked like a promising lead/real function name — it wasn't; don't chase engine-internal `K2_*` names surfaced in incidental error text, they don't necessarily map to a callable `find_node_types` entry.)

**The splice-placement decision:** `TickBrain`'s very first existing node is `SetDetectionMeter` (this tick's freshly-computed fill/decay value being written to the member var). The new "update the decal" logic needed both `AlertState` and `DetectionMeter` as CURRENT-tick values. Splicing the new `GetControlledPawn → Cast → UpdateVisionConeParams(GetAlertState, GetDetectionMeter)` chain in **before** `SetDetectionMeter` (i.e., at the very front of the function, the "obvious"/laziest insertion point right after `FunctionEntry`) would have read `DetectionMeter` **one tick stale**, for the exact reason already documented in the Step 3 "pure node re-evaluation" gotcha (`unreal-mcp-blueprints` skill): a pure `Get` node re-evaluates fresh at each consumption point in exec order, so reading it BEFORE the tick's `Set` gets the OLD value, not the one this tick just computed. **Fix: spliced right AFTER `SetDetectionMeter` instead** (`break_pins` the `SetDetectionMeter.then → IfThenElse.execute` link, insert the new chain between them, reconnect the tail) — guarantees `GetDetectionMeter` downstream reads the just-written value. `AlertState` may still be transition-frame-stale by one tick if it's set later in the same function (not chased further — imperceptible for a cosmetic decal, self-corrects within one frame). **Generalizes: before picking WHERE to splice new logic into an existing per-tick function, check whether the values you're about to read are computed earlier in that SAME function — if so, splice after that computation, not reflexively at the front.** Verified correct live in PIE (not just by compiling clean): teleported the player into the Watcher's cone and watched `AlertState` 0→2 and `DetectionMeter` 0→1 progress through the exact same exec path the splice now sits in, proving both that the values are live-fresh and that the new Cast doesn't silently dead-end the function (a failed `Cast` would have left `CastFailed` unconnected and permanently blocked all downstream state-transition/movement logic every tick — worth an explicit regression check any time a `Cast` is inserted into a mandatory, must-always-continue exec chain).

## 2026-07-02 (styling session, verification pass) — CONFIRMED BUG, SILENTLY WRONG FOR AN ENTIRE SESSION: `LightComponent.lightColor` is a 0–1 clamped `LinearColor`, NOT a 0–255 `FColor` — every light color set this session as `{r:255,g:158,b:64,a:255}`-style values silently clamped to pure white

**The mistake:** every `PointLight`/`RectLight` color this entire session was set using the same 0–255 byte-range pattern proven correct for OTHER color-bearing fields elsewhere in this project (e.g. `WorldOutliner`-style FColor fields on some components). Every one of those `set_properties` calls returned `{"returnValue": true}` — no error, no warning — so each was trusted without a readback. **They were all wrong.** `ObjectTools.list_properties` on a real `LightComponent` shows `lightColor`'s actual schema: `{"type":"object","title":"LinearColor","properties":{"r":{"minimum":0,"maximum":1}, ...}}` — a **HDR-capable LinearColor, hard-clamped 0.0–1.0 per channel**. Sending `{r:255, g:158, ...}` doesn't error; every channel independently clamps to its max (1.0), silently producing pure white `(1,1,1,1)` regardless of the intended tint. **Caught only by accident**, during an unrelated verification pass, when a `get_properties` readback on `PointLight_HouseD_Window_MC` showed `lightColor: {r:1,g:1,b:1,a:1}` instead of the warm tone that had been sent.

**Blast radius: 8 lights, the entire session's worth of "warm lamp glow" / "cool CRT blue" mood lighting** — the home-base room's `RectLight_HomeBase_CRT` (TVGlow) + 2 `PointLight`s (LampWarm, WindowGlow), Maple Court Pool D's streetlight (LampWarm), and all 4 house window lights (WindowGlow) were ALL rendering as neutral white this entire time, not the intended night-mood palette tones — a real, meaningful styling regression that had gone completely undetected across many "verified via screenshot" passes, because a scene lit by white lights isn't visually alarming the way a totally-dark or garishly-wrong-hued scene would be — it just quietly loses the warm/cool contrast that was the entire point.

**Fix:** re-set every affected light's `lightColor` using the SAME 0–1 `LinearColor` values already read from `DA_ArtPalette` earlier in the session (`LampWarm={1,0.62,0.25,1}`, `WindowGlow={1,0.78,0.42,1}`, `TVGlow={0.3,0.55,1,1}`) — i.e., the exact values the material system (`M_GreyboxToon`, `M_CRT`) had been using correctly the whole time, since material `VectorParameter`s use the same 0–1 convention and I'd always used it there. **Verified fixed via `get_properties` readback (no longer `1,1,1,1`) AND a fresh `CaptureViewport`** — the CRT's key light visibly changed from a neutral glow to a distinct blue pool of light on the home-base room floor.

**Generalizes, and is now a standing check:** `ObjectTools.list_properties` (or a `get_node_type_pins`-style min/max check) on the SPECIFIC component/property BEFORE assuming a color convention — this project has at least two different color conventions in play (0–255 `FColor` on some fields, 0–1 `LinearColor` HDR on others, confirmed for `LightComponent.lightColor` specifically) and neither the property name nor a successful-looking `set_properties` call distinguishes them. **A `set_properties` call returning `true` is NOT proof the value landed as intended — silent clamping is a real failure mode, not just silent rejection.** For anything visually load-bearing (colors, especially), always `get_properties` readback immediately after set, the same discipline already established for text/enum values. A useful tell in hindsight: FColor's channel range in a schema dump reads `0–255`/no explicit range (byte), LinearColor explicitly shows `"minimum":0,"maximum":1` in its JSON schema — that's the fast way to tell them apart before writing, not after.

## 2026-07-02 (styling session, HUD) — Step 7 HUD built via UMGToolSet end-to-end; several new widget-specific DSL/node gotchas, and BeginPlay DOES fire (a prior "maybe it doesn't" concern is resolved for PIE)

Built `WBP_HUD` (loudness bar, at-risk/banked score, neighborhood alert `?`/`!`, air bar, hide pip) fully via `UMGToolSet` + `BlueprintTools`/DSL, per `07_Movement_And_UI.md` §3 with names corrected against `CANON.md`'s drift table (`TeamScoreBanked`/`IndividualScoreAtRisk`/`AlertLevel`/`CurrentLoudness`, not the doc's stale short names). New widget-authoring gotchas, all found via trial-and-error against real tool errors (never guessed silently):

1. **A widget blueprint's OWN member-variable accessor prefix is `Variables|<BlueprintName>|Get<Var>`, not `Variables|Default|Get<Var>`** — the pattern used successfully all session for every OTHER Blueprint class. `Variables|Default|GetLoudnessBar` errors "does not exist"; `Variables|WBP_HUD|GetLoudnessBar` works. Likely because widget "variables" here are actually named child-widget bindings (`ToggleWidgetAsVariable`), not plain `add_variable` members — the DSL's accessor-name resolution keys off the owning Blueprint's name specifically for this category. Check `find_node_types(graph, "Variables|")` on the ACTUAL widget graph before assuming the `Default` prefix carries over from regular-Blueprint work.
2. **UMG's plain "Text" widget (`/Script/UMG.TextBlock`) SetText/SetColorAndOpacity show up under DISAMBIGUATED type_ids you have to pick correctly among near-duplicates**: `Widget|SetText(Text)` (bare "Text" widget = TextBlock) vs `Widget|SetText` (bare, no suffix — actually `RichTextBlock`, confirmed by its self-pin reading "Rich Text Block Object Reference" when checked via `get_node_type_pins`) vs `Widget|SetText(TextBox)`/`(EditableText)`/etc. **Always verify via `get_node_type_pins` — read the `self` pin's type name — before trusting a type_id name alone**; "no suffix" is not "the plain/default one," it's whichever widget class happened to register first. Same family: `Class|Text|SetColorandOpacity` (not `Class|TextBlock|...` — the friendly class name in this node-DB is "Text", matching the palette label, not the C++ class name).
3. **Building "LABEL: value" HUD text is much easier as two separate TextBlocks (static label + dynamic value) than one concatenated string.** No `Concat_StrStr`/plain-text-`+` node was found registered under obvious names; rather than sink time into discovering Unreal's actual Text-concatenation node, split every "prefix + live number" HUD element into a static label TextBlock (set once, never touched again) plus a value-only TextBlock updated via `Utilities|Text|ToText(Integer)` straight into `SetText`. Simpler graph, zero string-building risk, and arguably better UMG practice anyway (independent Bind-ability, matches the source doc's own "many small `WBP_` widgets" philosophy).
4. **`ProgressBar.SetPercent`'s param is `Percent`, not the property name `InPercent`/`percent`** you'd expect from the property JSON schema — property names (used with `ObjectTools.set_properties`) and FUNCTION parameter names (used with `write_graph_dsl`/`create_node` keyword args) are DIFFERENT namespaces that don't always match 1:1. Same for `Widget.SetVisibility`'s enum param, which is `InVisibility` (property is `visibility`). Verify function param names via `get_node_type_pins`, don't reuse the property-schema name blind.
5. **`Class|X|SetY` type_ids returned by `find_node_types` can report a misleadingly bare `type_id` field back from `get_node_type_pins`/`get_node_infos`** (e.g. `Class|ProgressBar|SetFillColorandOpacity` node's own `.type_id` reads back as just `|SetFillColorAndOpacity`, prefix stripped) — this is cosmetic/how the tool reports self-class setters, not a sign your `create_node`/DSL call used the wrong type_id. Don't "fix" a call that already succeeded because the echoed type_id looks different from what you passed.
6. **A Blueprint can have only ONE `Event BeginPlay` node per graph** (it's a single-dispatch virtual-function override, NOT a repeatable delegate-bound event like `EventTick` felt like it might be) — `create_node("AddEvent|EventBeginPlay", ...)` on a graph that already has one **returns/relocates the EXISTING node** (confirmed via `get_node_infos`: same existing "then" connections, just moved to the new requested position) rather than erroring OR creating a second one. Always `find_nodes`+`get_node_infos` to check for an existing singleton-event node before assuming `create_node` made you a fresh one — verify by checking whether its output pins already have connections you didn't just make.
7. **To add new logic to an existing `EventBeginPlay` chain without touching/understanding its full existing content: `break_pins` the ONE link right after the event node, splice your new chain in between, then reconnect your chain's tail to whatever the event used to connect to directly.** This is the safe "insert at the very front" pattern — you don't need to trace the rest of the chain at all, and you provably can't break anything downstream since its first input is unchanged, just now sourced from your tail instead of the event directly.
8. **Widget UMG `(event EventTick ...)` DSL sugar fails** with `AddEvent|EventTick does not exist` — for `UserWidget` graphs specifically, Tick is namespaced `AddEvent|UserInterface|EventTick` (unlike a regular Actor/Component graph where bare `EventTick`/`EventBeginPlay` DSL sugar works per the blueprints skill's existing guidance). The DSL's `(event Name ...)` shorthand doesn't take a category-qualified name — fall back to raw `create_node`/`connect_pins` for any event whose `AddEvent|...` type_id has a category segment.
9. **Verified BeginPlay DOES fire reliably in PIE** — added a `PrintString` immediately after a (possibly-stale-bytecode-suspected, see the `BP_PlayerController` cold-load entries below) existing `EventBeginPlay`, and `LogsToolset.GetLogEntries` showed it firing with zero `Accessed None`/runtime-error entries across the whole PIE session afterward (confirms the new HUD-creation chain, and by extension the chain it was spliced in front of, both execute). **This is PIE evidence only, not cold-disk proof** — the project's own established rule (only a fresh process tests on-disk truth) still applies; a cold standalone relaunch is the next honest check if the "cold load" symptom resurfaces.

## 2026-07-02 (costume integration session) — CRITICAL: a stub function created by another session as a Custom Event (not a plain function graph) silently resists `add_object_function_param`/`write_graph_dsl` edits forever, even though every call reports success and compiles clean

### The finding
`BP_CostumeComponent.Server_EquipCostume` had been stubbed by an earlier (different) session with a signature that LOOKED like a normal function (`read_graph_dsl` printed `(fn Server_EquipCostume () (|ExecuteUbergraphBPCostumeComponent 0))`). Unreal compiles every Custom Event as a thin dispatch stub into one combined `ExecuteUbergraph_<ClassName>` function — which is EXACTLY the body every failed edit kept reverting to, because compiling regenerates that stub fresh from the *real* Custom Event node's actual (unparameterized) signature in `EventGraph`, discarding whatever the function-editing tools had just written to a synthetic/transient view.
**Diagnosis recipe if this happens again:** if `add_function_param`/`write_graph_dsl` on a "function" report success but a subsequent `compile_blueprint` + `get_node_infos` on the ACTUAL entry node shows your change reverted, immediately check `BlueprintTools.list_graphs(blueprint)` — if the function name is absent from that list, it's a Custom Event, not a function, and none of the function-editing tools will ever stick.
**Fix:** find the real node (`AddEvent|Custom|<Name>` type, discoverable via `get_node_infos` once you know roughly where it lives) and `delete_node` it, then `add_function_graph` to create a genuine replacement with the same name. **Gotcha while doing this:** immediately after `delete_node`, `add_function_graph` with the exact same name still auto-suffixed to `_0` (name still reserved) — one `compile_blueprint` in between let the deletion fully register, after which the exact original name was free again. Once rebuilt as a real function, `add_object_function_param` + `write_graph_dsl` behaved normally and the parameter survived every subsequent compile.

### Related, smaller finding: `Class|X|Y` self-context pin ORDER for a cross-object call is `(self, ...dataArgs)`, not `(...dataArgs, self)` — despite property Setters being the other way around
Confirmed again this session: setter-style nodes for Blueprint VARIABLES (`Class|BPPlayerState|SetAir`, etc.) take positional args as `(value, self)` — matching `Class|CharacterMovementComponent|SetMaxWalkSpeed(value, self)` from earlier sessions. But a plain cross-object FUNCTION call (`Class|BPCostumeComponent|ServerEquipCostume`) takes `(self, dataArg)` — self FIRST. There is no way to guess which convention applies from the node's name alone; always call `get_node_type_pins` before writing a multi-arg cross-object call, exactly as the project's existing "always check pins for CallFunction nodes" rule already says — this is a second, independent reason that rule exists (not just the "extra self pin swallows position 0" issue documented earlier), so don't assume all self-inclusive nodes share one fixed argument order.

## 2026-07-02 (movement/hiding/swimming session) — `BP_BushHide` overlap testing: an hour lost to a wrong Z coordinate that LOOKED exactly like a collision bug — always re-`trace_world` the ground height at a NEW X/Y before reusing an old Z value

### The finding
Teleporting the player into a freshly-built `BP_BushHide.HideVolume` (a `QueryOnly`/`OverlapAllDynamic` BoxComponent — the same profile already proven to work fine for `BP_PoolVolume`/`BP_StashZone`) produced a bizarre, highly reproducible symptom: `bIsInBushVolume` stayed `false`, and `get_actor_transform` showed the capsule had been physically shoved sideways by a suspiciously exact amount every time (`box half-extent + capsule radius + ~1.125 skin`, e.g. `75+34+1.125=110.125`) — the shove direction always matched whichever axis had the SHALLOWEST penetration (a textbook minimum-translation-vector depenetration). This reproduced at four different X/Y target locations, survived even setting the box's `collisionEnabled` to `NoCollision` on the LIVE instance (a red herring — reflection-based property writes on a running physics body don't call `RecreatePhysicsState()`, so the old collision silently stayed active), and looked exactly like "this box is secretly blocking, not overlapping." **The actual cause: every single test used a stale/copied Z coordinate from a DIFFERENT location's ground-height trace** (reusing `PlayerStart`'s Z, or an earlier spot's already-traced Z, at a NEW X/Y without re-tracing) — the capsule was landing with its top a couple units UNDER the actual ground surface at the new spot, and it was **the ground**, not the bush, driving the depenetration; the bush's box (whose Z-span happened to closely match the same embed region) just made the resulting push look bush-shaped. ## 2026-07-02 (movement/hiding/swimming session) — Building `BP_BreathComponent`: five `write_graph_dsl` node-wiring gotchas, plus a real logic bug caught only by live PIE (`bIsInWater` vs `bIsSubmerged`)

### `(event EnhancedInputActionIA_X ...)` DSL syntax can only be used to relate to an EXISTING Enhanced Input event node — it cannot CREATE one
Trying to add brand-new `IA_Dive`/`IA_SwimUpDown` handlers via `write_graph_dsl`'s normal `(event EnhancedInputActionIA_Dive (...) ...)` form (which is how `read_graph_dsl` prints existing ones) fails: `AssertionError: AddEvent|EnhancedInputActionIA_Dive does not exist`. **Fix:** create the node the low-level way — `create_node` with `type_id: "Input|EnhancedActionEvents|IA_<ActionName>"` (find it via `find_node_types` filtered to `EnhancedActionEvents|IA_<Name>`), then `get_node_infos` on the new node to see its real exec pins (`Triggered`/`Started`/`Ongoing`/`Canceled`/`Completed`, plus a plain-typed `ActionValue` pin — Boolean for a Digital action, Float for Axis1D, no struct-unwrapping needed), then `connect_pins` by hand to a `CallFunction|YourHandler` node also made with `create_node`. This is the same "must delete+recreate with the correct `Input|EnhancedActionEvents|IA_X` type" rule from the original input-repair session, now confirmed to also apply to first-time creation, not just rebinding.

### `CallFunction|X` nodes for a custom Blueprint function sometimes expose an explicit "self" input pin that swallows your first positional arg
`(CallFunction|TickBreath DeltaSeconds)` (one arg, matching the function's one parameter) failed with `Could not connect pin DeltaSeconds to self` — `get_node_type_pins` showed the real pin order was `(execute, self, DeltaSeconds)`, i.e. TWO input pins beside exec, so the single positional arg bound to the wrong one. This contradicts the `Move`/`Aim` precedent from the ThirdPerson template (`(CallFunction|Move Axis_X Axis_Y)` — no self needed) — those nodes were built by the original `create_node`/`connect_pins` era of this project, not this DSL, so their "no self pin" shape isn't representative of what fresh `CallFunction` nodes for NEW functions look like. **Rule going forward: always call `get_node_type_pins` on a `CallFunction|X` node type before trusting positional args, and default to explicit keywords (`:self self :ParamName value`) for any freshly-added custom function** — cheap insurance against this exact silent-shape-mismatch class of error. Same fix applied to `Pawn|Input|AddMovementInput` (4 real input pins: self/WorldDirection/ScaleValue/bForce — the Move() function's 2-arg call was, again, pre-DSL legacy shape).

### Sequential top-level `switch` statements in one event/function body raise a false "unreachable code" error — `switch` has no single unified continuation pin the way `if` does
Adding a second, unrelated `switch` statement after an existing one in `EventOnMovementModeChanged` (checking `PrevMovementMode` right after already checking `NewMovementMode`) failed to write at all: `Unreachable code after branch/return`. A `switch` node's cases each dead-end on their own case pin — there's no single "next" pin for a following top-level statement to chain from, so the DSL compiler (correctly, if confusingly) refuses to treat anything after it as reachable. Adding an empty `(:Default)` case didn't help (this enum's switch has no Default pin — `SwitchOnEMovementMode`'s cases are exhaustive: `MOVE_None/Walking/NavWalking/Falling/Swimming/Flying/Custom`, no catch-all). **Fix: don't chain two switches — use `if (== EnumVar (Utilities|Enum|LiteralenumETypeName "EnumValue")) ... )` instead**, chaining as many `if`s in a row as needed (this form DOES support a normal single continuation, proven throughout this session in `TickBrain`/`ApplyMovementTuning`). Find the literal-maker node type via `find_node_types` filtered to the enum name (e.g. `EMovementMode` → `Utilities|Enum|LiteralenumEMovementMode`), and note the quoted-string-literal-as-enum-value trick from the DSL docs applies as the ARGUMENT to that literal node, not directly as an operand to `==`.

### Real bug caught only by live-PIE property inspection, not by clean compiles or code review: `TickBreath` drained Air based on `bIsInWater` instead of `bIsSubmerged`
Wrote `BP_BreathComponent.TickBreath` gating the drain/refill branch on `Class|BPPlayerState|GetIsinWater` — compiled clean, DSL round-tripped correctly, looked right on paper. Tried to directly poke `bIsSubmerged`/`bBreathCritical`/`bDetained` (all `RepNotify` bools) on a running `BP_PlayerState_C_0` instance to drive a test scenario without real input — every attempt failed with `the following properties could not be set: <name>`, while the exact same properties read back fine via `get_properties`. ## 2026-07-02 (movement/hiding/swimming session) — `ObjectTools.set_properties`: to create a NEW instanced sub-object inline (e.g. an `InputModifier` on an `InputMappingContext` mapping), pass the **class path** as the `refPath` value directly — do NOT wrap it as `{"instance": "<ClassPath>"}`

### The finding
Adding `IA_SwimUpDown`'s negative-axis key mapping (`Left Ctrl` → `-1`) to `IMC_Default` needed a brand-new `InputModifierNegate` instance in the mapping's `modifiers` array (existing modifiers on other mappings are pre-existing instanced sub-objects like `IMC_Default.IMC_Default:InputModifierNegate_0`, created once, long ago, presumably in-editor). The tool's own schema doc says "For instanced sub-object properties, pass a class path as the instance member" — read literally, this means wrap it as `{"instance": "/Script/EnhancedInput.InputModifierNegate"}`. **That produced a silent no-op**: `set_properties` returned `true`, but a follow-up `get_properties` showed the modifiers array element as the literal string `"None"` — no new object was created, no error was raised either.

The tool detects the path resolves to a *class* rather than an object instance and auto-instantiates a brand-new sub-object of that class (owned by the target asset), returning it as a real new instance (confirmed: `IMC_Default.IMC_Default:InputModifierNegate_1` appeared after save, verified via `get_properties` immediately after and again via a **cold standalone process** reading `GetAll InputMappingContext DefaultKeyMappings` from disk).
**Generalizes:** whenever an MCP tool's schema description mentions a special JSON shape for "creating" something inline, don't trust the described key name at face value — verify with a `get_properties` (or equivalent) round-trip immediately after the call. A `true` return and no error do NOT mean the operation did what you intended; this is the same family of trap as the `set_pin_value`-doesn't-mark-dirty lesson below (success signals that don't mean success).

## 2026-07-02 (post-MCP-reconnect session) — CRITICAL, SILENT-DATA-LOSS RISK: `BlueprintTools.set_pin_value` does NOT mark the containing package dirty, so `save_assets` silently skips it — a pin edit can affect live runtime behavior for an entire session while never reaching disk

### The finding
Fixed a verified, high-value bug (`TickBrain`'s chase `MoveToActor` `AcceptanceRadius=120` → `55`, see the caught/detain entry below) using `set_pin_value` on 3 duplicate nodes, then `compile_blueprint` (succeeded), then `save_assets([])` (returned `true`). **The fix demonstrably took effect at runtime** — re-tested live in PIE multiple times and the Watcher's chase behavior genuinely changed (walked to ~69 units instead of stopping at 120+). **The fix:** re-apply the same change via a full `write_graph_dsl` rewrite of the containing function instead of a surgical `set_pin_value`. Immediately after, `is_dirty` returned `true`, and `save_assets` + a `git status` check confirmed the file actually changed on disk this time. **`write_graph_dsl` reliably marks its target dirty; `set_pin_value` (at least for this pin/node type) does not.**
**Generalizes to a new, permanent verification habit: after ANY `set_pin_value` call (not just `write_graph_dsl`), check `AssetTools.is_dirty(path)` before trusting `save_assets`'s return value, and/or check `git status`/file mtime after saving.** A `save_assets() == true` return is NOT proof anything was written — it can mean "nothing needed saving" just as easily as "saved successfully," and there is no way to tell the two apart from the return value alone. This is a more insidious version of the project's existing "verify against cold on-disk state" rule: previous instances of that rule were about the EDITOR's dirty in-memory state vs. disk; this one is about the MCP TOOL's own reported success vs. disk, which is a new, narrower trap — the tool didn't lie about compiling or even about "saving," it just had nothing to save because the earlier edit never registered as a change worth saving.

## 2026-07-02 (post-MCP-reconnect session) — CRITICAL: `write_graph_dsl` on an event ALREADY PRESENT in a shared EventGraph accumulates orphaned nodes instead of cleanly replacing them; the live exec chain is still correct (rewired to the newest nodes) but the graph balloons with dead nodes on every re-edit

### The finding
Iteratively editing `BP_SensorLight`'s `EventActorBeginOverlap` handler in-place (4 successive `write_graph_dsl` calls, each adding one more diagnostic line) grew the shared `EventGraph` from ~20 nodes to 100+. Confirmed via `get_node_infos` on the `Event` node itself: `OtherActor`'s output pin had **5 separate consumers** (one `ActorHasTag` call per edit generation), and the event's single `then` exec pin was correctly repointed to only the NEWEST chain each time — so the graph was *functionally* still correct (only the latest logic actually runs), but stuffed with 80+ dead, disconnected nodes from every earlier edit. **This is unlike editing a plain FUNCTION graph** (`(fn Name ...)`), which cleanly full-replaces on every `write_graph_dsl` call — confirmed by contrast, e.g. `BP_AlertDirectorComponent:PollWatcherAndPush` and `:NotifyDetection` stayed at a small, stable node count across many re-edits. **The difference is specifically about editing an EVENT that already exists in a graph shared with other events** (`EventBeginPlay`/`EventTick` also live in `BP_SensorLight`'s `EventGraph`) — the tool seems to append a new node cluster and only rewire the single-owner exec/delegate pins, not diff-and-replace the event's whole body.

**The bug (would have shipped silently without live testing):** `Server_BankAtRisk(PS)` computed `IndividualScoreBanked += atRisk` (correct — PlayerState side matched expectations in testing), but the parallel `GameState.TeamScoreBanked += atRisk` / `TeamScoreAtRisk -= atRisk` came out **wrong** — `TeamScoreAtRisk` stayed stuck at the pre-bank value and `TeamScoreBanked` stayed 0. `get_node_infos` proved the *graph wiring* was 100% correct — one `GetIndividualScoreAtRisk` node, its single output wired to all three consumers (`IndividualScoreBanked` add, `TeamScoreBanked` add, `TeamScoreAtRisk` subtract), exactly matching the DSL source and the "bind once, reuse" rule from `get_graph_dsl_docs`.

**Root cause:** in my original statement order, `SetIndividualScoreAtRisk(PS, 0)` (the reset) ran **before** the GameState math (which lived inside a `CastToBP_PlayerGameState` `:then` block positioned later in the function). A **pure** node (no exec pins — a plain member-variable getter) has no cached/materialized value: Blueprint re-evaluates it **fresh at each consumption point**, not once at authoring time. So `SetIndividualScoreBanked` (which executes early, before the reset) read `atRisk` correctly; but `SetTeamScoreatRisk`/`SetTeamScoreatRisk` (executing later, after the reset already zeroed the source variable) re-pulled the SAME `GetIndividualScoreAtRisk` node and got **0** instead of the frozen original value. One DSL node ≠ one evaluation — "bind once, reuse" (from the docs) only guarantees you don't author a duplicate node; it does **not** freeze a pure read against a later Set on the same property within the same function.

**Fix:** reorder so **every** consumer of a soon-to-be-reset variable's pure read happens **before** the reset statement, not just the first one. Both `Server_BankAtRisk` and `Server_LoseAtRisk` had this bug (identical shape: read `IndividualScoreAtRisk` → reset it early → later re-derive a GameState delta from the same read). Fixed by nesting the reset **last**, inside the same exec branch as the GameState update, so the reset physically cannot run before every consumer of the old value has fired.

**Generalizes to:** any function that (a) reads a mutable variable, (b) resets/overwrites that SAME variable, and (c) needs the OLD value in more than one place — the fix is always "do all reads-dependent-math first, reset last," never "bind once and trust it." This will matter again for Step 5 (AlertDirector heat/detain resets) and Step 2 finish (loudness resets) — watch for this exact shape.

**Why this was caught:** direct PIE property inspection (`ObjectTools.get_properties` on the live `PlayerState`/`GameState` instances during a running PIE session, via `set_actor_transform` teleports into the pool then the stash zone) — not just "does it compile." A clean `compile_blueprint(warnings_as_errors=true)` and a correct-looking `get_node_infos` dump both said "fine"; only reading the actual runtime numbers exposed the bug. Reinforces the project's core verification discipline (see `unreal-mcp-session` skill §3) one level deeper: even *logic*, not just *assets*, needs a live runtime check, not just a clean compile.

### DSL: member variables need explicit getters; latent/multi-out nodes need keyword pins
In `write_graph_dsl`, a bare `MyVar` only resolves to locals/params — a member variable must be `(Variables|Default|GetMyVar)` or you get "Undefined variable". And positional args bind in pin order, which put a value on `self` for `Character|LaunchCharacter` (self is arg 0). Use **keyword args** (`:LaunchVelocity ... :bZOverride true`) for anything past the first data pin. Always `get_graph_dsl_docs` before first use.

### Enhanced Input mapping JSON shape (for `ObjectTools.set_properties` on an IMC)
`mappings` is an array of `{action: {refPath}, key: {keyName: "..."}, triggers: [], modifiers: [{refPath: "/Script/EnhancedInput.InputModifier<X>"}]}`.
