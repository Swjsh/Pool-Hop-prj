## 2026-07-03 (loot system build) — **Two new MCP gotchas: `Array|Add`/`AppendArray` fed by a variable-getter doesn't reliably write back through this MCP's node-creation path (needs an explicit follow-up `Set`), and newly-placed World Partition actors can spawn at `(0,0,0)` in PIE until the level is reloaded. Also: `PrimitiveTools.add_cube/sphere/cone/cylinder` adds a REAL persistent SCS component when called on a Blueprint's CDO — much simpler than the `SubobjectDataSubsystem` Python route for primitive meshes.**

**Gotcha 1 — array by-ref write-back is not reliable via MCP-created nodes.** Standard Blueprint pattern: wire a `Get MyArrayVar` node's output directly into an impure function's by-ref array parameter (e.g. `Array|Add`'s `TargetArray`, `AppendArray`'s `TargetArray`) and the compiler auto-detects "this ref traces back to a plain variable" and writes through to the actual variable memory — no explicit `Set` needed, this is extremely common in hand-authored Blueprints. **Built and wired this exact pattern twice tonight (loot pickup's `CarriedLoot` Add, GameMode's `BankedLoot` AppendArray) and the write-back did NOT persist either time** — `get_node_infos` confirmed the wiring was textbook-correct (Get→TargetArray, impure node, proper exec chain to a working `DestroyActor`/score-set afterward), the exec chain provably ran to completion, yet a live PIE property read showed the array unchanged. **Fix: add an explicit `Set<ArrayVar>` node immediately after the Add/Append, self=same target object, value=the same `Get` node's output (safe to fan out a pure node's output to a second consumer).** This is now a required extra step for EVERY array mutation built via MCP node creation in this project — don't trust the implicit by-ref write-back the way you would in the graphical editor. Confirmed fixed for the `CarriedLoot` case (live PIE re-test showed the item correctly present after the fix); the `BankedLoot` case has the identical fix applied and compiles clean but wasn't independently re-confirmed live due to a separate, compounding test-methodology issue (see below) — treat it as "same fix, same confidence," not "unverified."

**Gotcha 2 — newly-MCP-placed World Partition actors can read `(0,0,0)` in a PIE session instead of their authored transform.** Placed 3 `BP_LootPickup` instances via `SceneTools.add_to_scene_from_asset`, verified their transforms correctly via the PERSISTENT-level actor ref (matched what was set) — then started PIE and read the SAME actor via its `UEDPIE_0_`-prefixed PIE-world duplicate: `(0,0,0)`, not the authored position. This is presumably a WP streaming/registration gap for freshly-added external actors (same family as the already-documented "AI patrol points need the NavMesh rebuilt" and "SupportedAgents needs an editor restart" issues — WP/nav data has its own build/cache step separate from the plain asset save). **Fix: `SceneTools.load_level(same_level_path)` to force a reload before the next `StartPIE`** — confirmed this resolves it (re-checked the same actor's PIE-world transform after reload: matched the authored position). **Do this reload as a matter of course after placing/repositioning ANY actor via MCP, before trusting its transform in a PIE test** — cheap, and the alternative is a confusing "my object is at the origin for no reason" chase.

**`PrimitiveTools` adds a real persistent component directly on a CDO — a much simpler path than the documented `SubobjectDataSubsystem` Python recipe, for the specific case of a primitive-shape mesh.** Calling `PrimitiveTools.add_cube(actor=<CDO ref>, name=...)` (and presumably `add_sphere`/`add_cone`/`add_cylinder`) on a Blueprint's `Default__<Class>_C` CDO reference returned a `<Name>_GEN_VARIABLE`-suffixed component ref — the same naming signature the `SubobjectDataSubsystem` recipe produces, confirming it's a genuine persistent SCS component, not a CDO-only override. Tried the raw Blueprint asset ref first (`/Game/.../BP_Foo.BP_Foo`) — rejected ("not valid Actor for property 'actor'") — the CDO ref (`BP_Foo.Default__BP_Foo_C`, from `BlueprintTools.get_default_object`) is what's needed. **Use this for any primitive-mesh component need (a pickup's visual+collision mesh, a trigger-shaped mesh, etc.) before reaching for the full Python `SubobjectDataSubsystem` dance** — that recipe is still needed for non-primitive component types (skeletal meshes, widget components, custom mesh assets), but not for basic shapes.

**Also confirmed:** `add_function_param` only accepts primitives/basic structs (`bool int float byte name string text Vector Rotator Transform Vector2D LinearColor`) — no object-reference types, despite existing GameMode functions (`Server_BankAtRisk`, `HandleDetain`) clearly having `BP_PlayerState`-typed parameters (built in an earlier session via a mechanism not currently exposed, or since changed). **Workaround: splice new logic directly into an EXISTING function that already has the object parameter you need in scope**, using exec-chain-tracing (`find_nodes` + `get_node_infos`, follow `then` pins to the true unconnected terminal) to find a safe append point, rather than trying to create a new function with its own object parameter.

## 2026-07-03 (post-reconnect session) — **Swim CLOSED OUT: `MOVE_Swimming` confirmed on 2 pools live. A single teleport does NOT trigger a physics-volume re-check on the very next read — this cost a false "still broken" reading before a follow-up move revealed it works fine. Also: `bUseDefaultCollision` has NO Blueprint setter, so the self-heal targets `SetCollisionEnabled` instead — and it's genuinely more general as a result.**

**MCP reconnected clean this session** — `list_toolsets`/`IsPIERunning`/`get_current_level` all returned real, live data on the first real calls (not just a `ToolSearch` hit), confirming last session's diagnosis was right: the prior session's client just never got a live handle because the editor was down at its own startup; a plain Claude Code restart (with the editor+port already up) was the whole fix, no special recovery needed.

**The false alarm, and the actual mechanism:** teleported the player into the sandbox pool's `PhysicsVolume` bounds via `set_actor_transform`, immediately read `CharMoveComp.MovementMode` → got `MOVE_Walking`, THREE reads in a row, despite `get_actor_transform` confirming the pawn really was sitting deep inside the volume and `bWaterVolume=true` on the volume itself. Nearly wrote this up as "swim behaviorally still broken despite correct collision." One more probe first (per the debugging discipline — don't theorize past the evidence): sent a SECOND, small position update (a few units of Z) and re-read — `MOVE_Swimming` immediately. Repeated on a second pool (Maple Court A) with the identical two-step pattern: same result both times. **Mechanism: `AActor::UpdatePhysicsVolume`-style membership checks don't necessarily re-run on the exact frame a hard `SetActorTransform`/teleport lands — they need a subsequent transform update to trigger re-evaluation**, which is a non-issue for a real player (every tick of `AddMovementInput`-driven motion IS a transform update) but WILL fool a single-teleport-then-immediate-read test every time. **Generalizes: any MCP-driven "teleport then immediately read volume-membership/overlap state" test should do a teleport + one small follow-up nudge + THEN read, or it risks a false negative on the very mechanism it's trying to verify.** Also confirms the historically-recurring collision bug (`bUseDefaultCollision`) is genuinely still fixed — verified fresh via `bWaterVolume=true` + successful swim entry on 2 separate pools, not just an old claim.

**Self-heal for the recurring collision regression — redesigned mid-build once a real MCP limitation was found, and the result is better than the original plan.** The original plan (`12_Overnight_Session_Plan.md` §0) called for forcing `bUseDefaultCollision=False` on `BeginPlay`. `find_node_types` on a `Collision` filter turned up `Collision|SetCollisionEnabled` but **no setter for `bUseDefaultCollision` anywhere in the node catalog** — it's evidently not `BlueprintReadWrite` at all (an authoring-time-only flag), so no Blueprint graph can touch it, full stop. Pivoted to a mechanism that doesn't need to: `BP_PlayerGameMode.EventBeginPlay` now does `GetAllActorsOfClass("/Script/Engine.StaticMeshActor")` → cast each → compare its `StaticMeshComponent.GetStaticMesh()` against a new `WaterSurfaceMesh` object variable (set to `/Engine/BasicShapes/Plane`) → `SetCollisionEnabled(NoCollision)` on every match. A runtime `SetCollisionEnabled` call overrides whatever the initial body-setup produced regardless of what `bUseDefaultCollision` did at spawn — so this achieves the same functional goal (water never blocks the player) without needing write access to the property that's actually causing the historical regression. **Bonus: this is mesh-identity-based, not per-pool-wired** — it automatically covers all 5 current water surfaces (and any future ones) with zero per-pool setup, versus the originally-planned per-`BP_PoolVolume` reference wiring, which would have needed 5 separate correctly-paired variable assignments and doesn't exist as an owned relationship in the actual architecture (water surfaces are independently-placed `StaticMeshActor`s, not components of `BP_PoolVolume`).

**Verified as a real guard, not just a happy-path check (RED-PROOF pattern):** before trusting this, deliberately broke one water surface's `CollisionEnabled` to `QueryAndPhysics` on the EDITOR (persistent-level) actor, started a fresh PIE session, and read the LIVE PIE-world copy of that same component — it came back `NoCollision`, proving the `BeginPlay` logic actually corrects a broken state at runtime rather than merely leaving an already-correct one alone. This is the class of verification this exact bug's history has been missing every prior time it was "fixed."

## 2026-07-03 — **`Montage_Play` is a real workaround for the AnimGraph ceiling — it's a regular EventGraph function, not an AnimGraph node, so it's fully reachable via BlueprintTools even though the AnimGraph itself isn't.**

Given zero animations exist anywhere in the project for swim/vault/slide/crouch, and Control Rig can't be wired into the AnimGraph (confirmed 2026-07-03 earlier entry), the practical middle ground: wrap an EXISTING, tonally-adjacent AnimSequence into a new `AnimMontage` asset (`unreal.AnimMontageFactory` with `source_animation` set — same factory-based creation pattern used for materials elsewhere, works cleanly, no gotchas hit), then call `Animation|PlayAnimMontage` from a normal Blueprint function (`TryVault`, `StartSlide`) exactly like any other function call. This sidesteps the AnimGraph-node wall entirely, because Montage playback is a *procedural* pose override layered on top of whatever the AnimGraph outputs — it's driven from gameplay code, not graph wiring. Used `MM_Dash` (0.97s) as a stand-in for both slide-start and had `MM_WallJump` (0.47s) stand in for vault-launch — neither is a bespoke animation for the action, but both give real triggered motion instead of the character sliding/launching in a static pose. **This is the same trick documented for the AI Watcher's catch/detain flow conceptually (a system reacting via EventGraph triggers, not AnimGraph rewiring) — worth remembering as the general pattern whenever "the character needs to visibly react to X" and the AnimGraph itself is off-limits: look for an EventGraph-callable animation hook before assuming the whole feature is blocked.**

**Reused `ResolveSpeed` instead of building a parallel Tick system for the slide's momentum decay.** The character already had one per-tick speed-resolution function computing `MaxWalkSpeed` from sprint/crouch/swim state. Rather than add a second per-tick handler (risking two systems fighting over the same property, and Blueint only allows one `EventTick` anyway), the slide decay (`FInterpTo` from a boosted speed down to `WalkSpeed`, auto-clearing `bIsSliding` once it settles) was spliced into that SAME function as an early-exit branch, leaving the original sprint/crouch/swim logic completely untouched in the `else` path. **General pattern: before adding a new per-frame handler for a new movement state, check whether an existing per-tick function already owns the property you need to override — extending it is lower-risk than a second competing writer.**

**Deepening a pool's water is safe to do via Z-scale/reposition alone — it's the X/Y footprint changes that risk neighbor overlaps (already learned the hard way once this project, see the enlarged-pool-overlapped-the-bush entry).** Extended each pool's basin + physics volume ~130 units further down (bottom face moves, top/water-line Z stays fixed) with zero risk of colliding with fences/bushes/houses at the surface level, since nothing about the pool's outline in the XY plane changed. A useful distinction going forward: "make the pool feel bigger/deeper" splits into a low-risk Z move and a higher-risk XY resize, and they don't need to be bundled.

## 2026-07-03 — **SOLVED: the swim-collision bug that recurred 3+ times across this project's history. Root cause was `StaticMeshComponent.bUseDefaultCollision`, not the `collisionEnabled` property everyone (including two attempts earlier this same session) kept re-fixing.**

Full mechanism: every water surface (`MI_Water_{A,B,C,D}_MC` + sandbox `PoolWaterSurface`) uses the shared engine mesh `/Engine/BasicShapes/Plane`. That mesh asset's own `BodySetup.DefaultInstance` is hard-baked to `BlockAll`/`QueryAndPhysics` — a totally reasonable default for a generic plane primitive used all over the engine for walls/floors/etc. Every fix attempt (this session's first two, and per this file's own earlier entries, at least two from prior sessions) set the **instance-level** `StaticMeshComponent.BodyInstance.CollisionEnabled` to `NoCollision` — which is the *correct, normal* way to override collision on a placed instance, and it DID work... in the sense that reading the property back immediately in the editor always showed `NoCollision`, exactly as set. **But every one of those water surface components also had `bUseDefaultCollision = True`.** That flag makes `UPrimitiveComponent` ignore its own instance-level `BodyInstance` override entirely at *runtime* (`BeginPlay`/PIE-start physics-body creation) and pull collision straight from the mesh asset's `BodySetup.DefaultInstance` instead. Net effect: the instance override was real, verifiable, and **completely inert** — a plausible-looking fix that could never have worked, no matter how many times someone re-applied it, because the actual runtime physics body was never built from it in the first place.

**Why this took 3+ sessions across this project's history to catch, and why an in-editor property readback can never catch it:** reading `bodyInstance.collisionEnabled` on the placed actor in the editor (the verification method used every previous time, including my own first two attempts this exact session) shows the *instance's stated intent*, not what the runtime physics engine will actually use — those are two different data paths that only converge if `bUseDefaultCollision` is false. **The only way this bug was findable at all was a genuinely fresh PIE session's live property read** — checking the SAME component, mid-Play, and seeing `QueryAndPhysics` despite a saved-and-reloaded editor state reading `NoCollision`. That specific "editor says X, live PIE says NOT-X, on an asset nobody touched in between" signature is what pointed at "something is regenerating collision from a different source at BeginPlay" rather than "the fix didn't save" (which was the wrong hypothesis for 2 straight attempts).

**Generalizes to ANY future "the collision fix isn't sticking" mystery on a component using a shared/engine primitive mesh** (Cube, Plane, Cylinder, Sphere from `/Engine/BasicShapes/`): check `bUseDefaultCollision` FIRST, before re-touching `collisionEnabled` again. If it's `True`, the asset's own `BodySetup.DefaultInstance` collision (queryable via `mesh.get_editor_property("body_setup").get_editor_property("default_instance")`) is what actually governs runtime behavior, and any instance-level override is cosmetic until that flag is flipped to `False`.

**Also confirms/extends the earlier same-day finding**: `EditorActorSubsystem.get_all_level_actors()` returns **zero actors** while PIE is active (not just "doesn't see PIE-world actors" — it returns nothing at all, even for the persistent editor level). The direct MCP `SceneTools.find_actors`/`ActorTools.*` calls DO resolve during PIE, but only to the `UEDPIE_0_`-prefixed transient PIE-world duplicate — reads are trustworthy (reflects live game state, genuinely useful for exactly this kind of live diagnosis), but writes to those refs are lost the moment PIE stops. To make a PERSISTENT fix, PIE must be fully stopped first; there is no way to reach the editor's real persistent-level actors for writing while a Play session is running.

## 2026-07-03 — **The pool-water-blocks-swimming bug is now a THIRD confirmed occurrence — this is a recurring regression with an unknown root cause, not a one-off. A live human playtest caught what property-based "verification" missed twice already.**

## 2026-07-02 — **Adding a PERSISTENT component to a Blueprint: BlueprintTools has NO `add_component` — use the editor-Python `SubobjectDataSubsystem`. (This is how the vision cone finally shipped as a mesh, and how the decal/icon components were originally added.)**

BlueprintTools does variables/functions/events/nodes/graphs but **cannot add an SCS component** (confirmed by listing its tools). The path that works, via the editor Python console:
```python
subsys = unreal.get_engine_subsystem(unreal.SubobjectDataSubsystem)
bp = unreal.load_asset("/Game/.../BP_Foo")
handles = subsys.k2_gather_subobject_data_for_blueprint(bp)   # handles[0] = root
params = unreal.AddNewSubobjectParams()
params.set_editor_property("parent_handle", handles[0])
params.set_editor_property("new_class", unreal.StaticMeshComponent)
params.set_editor_property("blueprint_context", bp)
new_handle, fail_text = subsys.add_new_subobject(params)      # fail_text '' = ok
subsys.rename_subobject(new_handle, "VisionFanMesh")
comp = unreal.SubobjectDataBlueprintFunctionLibrary.get_object(
           subsys.k2_find_subobject_data_from_handle(new_handle))  # the SCS template
comp.set_static_mesh(mesh); comp.set_material(0, mat)
comp.set_editor_property("relative_scale3d", unreal.Vector(...))
comp.set_editor_property("relative_location", unreal.Vector(...))
comp.set_editor_property("relative_rotation", rot)
comp.set_collision_enabled(unreal.CollisionEnabled.NO_COLLISION)
unreal.EditorAssetLibrary.save_loaded_asset(bp, only_if_is_dirty=False)
```
Worked first try. Gotchas: the created object is named `<Name>_GEN_VARIABLE` (a template, not the runtime instance) — set properties on IT and they serialize into the BP; `compile_blueprint` after so **placed instances pick up the new component** (SCS components DO propagate to placed instances, unlike CDO material/visibility overrides); make the add idempotent by scanning existing handles for the name first (re-runs then reconfigure instead of duplicating). This is **editor-visible**, so you can iterate the transform with the cheap non-PIE `CaptureViewport` loop.

**Applied: the mesh vision cone that the decal couldn't do.** Decal projection bled onto walls (see prior entries); a **flattened Engine `Cone` mesh** as an SCS component doesn't — it's placed geometry. Recipe: `M_VisionConeFan` (Surface/Translucent/Unlit/TwoSided, `FanColor`+`FanOpacity` params), `relRot pitch=-90` lays the cone axis flat (local-X→world-Z, so `scale.x=0.05` flattens it to a slab; y=lateral, z=length), **`yaw=180` flips the apex to the eye** (apex-forward otherwise), `relLoc (150,0,-88)` drops it to the floor, no collision, parented to root → follows facing. Clean floor FOV cone, verified via non-PIE overhead capture. Two red herrings while iterating: (1) the flatten axis — flattening local-Z gives a thin vertical *blade*, not a floor fan (must flatten the axis that maps to world-Z after the pitch); (2) a persistent "mystery cyan square" in captures was just the **sandbox pool's translucent water surface**, not vision debris (an `identify by material` sweep confirmed only one actor used the vision material).

## 2026-07-02 — **THE CATCH/DETAIN LOOP WORKS — closing the last broken core mechanic. Verified live: patrol → detect → chase → catch at CatchRadius → detain → at-risk lost → 2s → respawn at stash → Watcher resumes patrol.** Resolves the "caught/detain unreachable in play" design gap (the old "3 candidate fixes need a human decision" flag) under the hands-off mandate.

**The design rule implemented:** catch fires when the Watcher is Suspicious-or-Alert AND the target is within the profile's own `CatchRadius` (120) — **independent of current sight**. This was necessary because of a cleanly-reproduced engine behavior: **AISense_Sight goes blind at point-blank range** (observed: the Watcher standing 68uu from the player — capsules touching — with `bCanSeePlayer=false` and the meter decaying; the eye-to-target trace ends inside the target's own capsule). The overlap-based catch was doubly dead: move completion leaves the pawn at AcceptanceRadius+capsules (~108-123uu), so capsule overlap (68uu) is never reached either. Distance-check-at-alerted is the standard stealth-game resolution and uses only numbers the profile already specs (`CatchRadius`, `DetainRespawnDelay`).

**A DSL structuring gotcha cost one broken iteration:** placing the catch block BEFORE the movement dispatch, gated as `(if (and (!= alert 0) (Utilities|IsValid target)) ...)`, silently killed the whole main exec path when the target was null (patrol froze at spawn again — caught by re-running the patrol check before the catch test, which is now the standing pattern: **after ANY TickBrain edit, verify patrol still runs BEFORE testing the new feature**). Avoid `Utilities|IsValid` inside `and` expressions in DSL — whichever overload the writer picks, the lowering broke main-path exec here.

## 2026-07-02 — **AI WATCHER FULLY ENABLED, HANDS-OFF: the "movement regression" was THREE stacked causes, not one — and the long-standing "needs a human Build Paths click" diagnosis was only ever a third of the story.** Both Watchers now verified patrolling their full loops + chasing + investigating, live in PIE, zero human clicks.

## 2026-07-02 — **ROOT CAUSE FOUND + FIXED: the "black screen / can't see the game" was TWO stacked lighting-authoring bugs, NOT a render/GPU/focus problem — and the long-standing "editor renders black when not OS-foreground" lesson partially mis-covered for it.** The user could not see ANY geometry in the viewport ("Lit" mode, only wireframes/gizmos on black); every MCP capture came back the same.

**The two bugs (both in saved level content, which is why restarts didn't help):**
1. **Degenerate exposure lock on the unbound night PP volume:** `AutoExposureMinBrightness=0` AND `AutoExposureMaxBrightness=0` both *overridden* (a nonsense clamp), on top of `AEM_Manual` metering with bias −1. With the project's `r.DefaultFeature.AutoExposure.ExtendDefaultLuminanceRange=True`, those fields are EV100 units — the combination crushed the entire exterior to *mathematically pure black*, not "dark night."
2. **Palette token misused as a light color:** `BP_Moon`'s DirectionalLight `lightColor` was set verbatim to `DA_ArtPalette.nightBlueKey` `(0.04, 0.07, 0.16)` — a token authored as an on-screen *surface/UI* color. As a *light* color it multiplies intensity: 1.8 lux × ~0.16 ≈ 0.3 lux of near-black navy. Effectively no moon at all. **Rule: palette tokens give lights their HUE (normalize max channel → 1.0), never their brightness — brightness lives on `intensity`.**

**The fix (verified visually via before/after captures):** on the unbound PP volume — `AEM_Histogram`, bias 0, min/max brightness overrides re-locked to EV100 = 0/0 (a *sane* lock: consistent stealth-readability exposure with no adaptation lag); on the moon — `lightColor=(0.25, 0.44, 1.0)` (nightBlueKey hue at full brightness), `intensity=3.0`. Result: full moonlit-blue night everywhere; home-base room (own +3EV bounded volume) unaffected and still correct.

**Why this evaded diagnosis for a day+ (the meta-lessons, more valuable than the bug):**
- **A plausible documented lesson became a thought-terminating explanation.** Every black capture since ~July 1 evening was attributed to the (real, but narrower) "editor renders black when not OS-foreground" lesson. That lesson is about `CaptureEditorImage` returning stale/throttled *window* frames — but `CaptureViewport` with an explicit transform does a real scene render regardless of focus, and it was black for a *different* reason. **When a known gotcha "explains" a symptom, check whether its mechanism actually covers the observation before filing it away.**
- **`CaptureAssetImage` thumbnails can serve a STALE CACHED image** — an early "the renderer is healthy, look, the material sphere renders" proof was unsound (thumbnails may predate the breakage). Don't use asset thumbnails as evidence about live scene rendering.

One hypothesis → one change → one test only works if the change actually applied.
- **What finally cracked it:** capturing the home-base room (which has its own bounded +3EV exposure-override volume + unlit emissive CRT) rendered *perfectly* while the exterior was black — same world, same renderer, same materials. ## 2026-07-02 — **A component's `RelativeLocation` is NOT always "offset from a sibling/parent" — check `rootComponent` FIRST, because for the actor's ROOT component, `RelativeLocation` effectively IS the actor's world transform.** Nearly "fixed" a non-bug by moving an actor 300uu.

**What happened:** investigating `BP_ItemPickup_QuietShoes` for the same collision-blocking bug class as the pool/bush fixes (previous entries), found `PickupVolume.relativeLocation` = `(300, 1700, 208)` — suspiciously identical to the actor's own world position. Assumed this was a copy-paste bug (someone pasted world coords into a relative-offset field) that would place the pickup's trigger volume ~3800uu away from its own visual mesh (`PickupMesh`, at relative `(0,0,0)`). Attempted to "fix" it to `(0,0,0)`.

**What actually happened:** `PickupVolume` is the actor's `rootComponent` (checked via `ObjectTools.get_properties(actor, ["rootComponent"])` — do this BEFORE assuming any position bug). For a root component, there is no parent to be "relative to" — `RelativeLocation` on the root directly defines (or heavily constitutes) the actor's own world transform. So `PickupVolume.relativeLocation == ActorWorldLocation` was never a bug — it's the **expected, correct** relationship. `PickupMesh` (a genuine child, attached to the root) was ALREADY correctly co-located with `PickupVolume` by construction (its own relative `(0,0,0)` places it exactly at the root's position) — there was no mismatch to fix in the first place.

**The damage and the recovery:** the "fix" attempt (`set_properties(PickupVolume, {relativeLocation: {0,0,0}})`) partially succeeded — only the X component actually applied (see the `set_properties` partial-struct-write gotcha, same entry family), moving the WHOLE ACTOR from world X=300 to X=0 (since moving the root moves the actor). Caught immediately via `get_actor_transform` readback (not just the `set_properties` return value), and fully reverted with `ActorTools.set_actor_transform` back to the actor's original, correct `(300, 1700, 208)`. Re-verified after reverting. No lasting damage, but this could have shipped a genuinely displaced actor if the revert-check habit had been skipped.

**Generalizes — do this check BEFORE touching ANY component's transform for a suspected "position mismatch" bug:** `ObjectTools.get_properties(actor, ["rootComponent"])`, compare its `refPath` against the component you're investigating. If they match, that component's `RelativeLocation` is the actor's own placement — don't reason about it as "this should match a sibling's relative offset," and if you DO need to reposition it, use `ActorTools.set_actor_transform` on the ACTOR (which is the semantically correct operation), not `set_properties` on the component's `relativeLocation` (which is the same underlying value but invites exactly this category of confusion).

## 2026-07-02 — **`ObjectTools.set_properties` can silently apply only PART of a single struct-valued property**, not just silently drop whole properties from a multi-property call (see the entry below) — even a single `relativeLocation: {x,y,z}` write can land only `x` and leave `y`/`z` unchanged, with the call still returning `true`

Found while investigating the `PickupVolume` position (see the root-component entry above): two SEPARATE, isolated `set_properties` calls, each specifying `relativeLocation: {x:0, y:0, z:0}` as the ONLY property in the call, each returned `true`, and each time a follow-up `get_properties` showed only `x` had actually changed — `y` and `z` silently stayed at their original values, identically, both times (not a random/flaky result — the same partial outcome twice in a row). This is a more granular version of the already-documented "multi-property call can silently apply only some properties" gotcha — this time it's WITHIN a single struct-typed property, not across multiple top-level ones.

**Generalizes:** for any struct-valued property write (`Vector`, `Rotator`, `Transform`, etc.) that matters, re-read ALL of its sub-fields after the write, not just a couple. A `true` return, and even a partially-correct-looking readback, is not proof the whole struct landed. If a field doesn't stick after a retry, stop trying to fight it via `set_properties` — reconsider whether you're editing the right thing at all (see the root-component entry above: in this case the "stuck" fields were actually a signal something more fundamental was misunderstood, not just a flaky write).

## 2026-07-02 (NS_PoolSplash) — **`connect_pins` does NOT support exec-pin fan-out** — calling it a second time on the same output exec pin silently REPLACES the existing connection instead of adding a parallel one, unlike the graphical Blueprint editor where dragging a second wire off one output pin is completely normal. Nearly shipped a broken pool.

**What happened:** wanted to add a new "spawn splash VFX" call that fires ALONGSIDE `BP_PoolVolume`'s existing `EventActorBeginOverlap → IfThenElse_0` scoring logic, without touching that existing chain at all — reasoning that since real Blueprint exec output pins support multiple outgoing wires (both fire, in wire order, when the event triggers), I could just `connect_pins` a NEW target onto the SAME already-connected output pin as a safe, zero-risk addition. **This assumption was wrong for this MCP tool specifically.** After the call, `get_node_infos` on the event node showed `then` connected ONLY to my new node — the original connection to `IfThenElse_0` was gone. Re-checking `IfThenElse_0.execute` directly confirmed it: `connected_pins: []`, completely disconnected. **This would have silently killed the entire pool's scoring path** (`PoolScoreComp.Server_EnterPool` and everything downstream) had it shipped un-caught.

**Fix, and the correct general pattern:** treat `connect_pins` as **single-connection-per-output-pin, always** — never assume a second call adds a parallel branch. To add new logic that should fire alongside (not instead of) an existing chain, **splice linearly**: `break_pins` the existing link, connect the event's output to your NEW node's execute input, then connect your new node's OWN "then" output to the ORIGINAL target. (`Event → YourNewNode → OriginalTarget`, not two independent listeners on `Event`.) This is the exact same "splice at the front" pattern already used successfully elsewhere this session (the vision-cone `TickBrain` splice, the `AlertState` mirror splice) — the difference here is realizing it's not just the SAFEST option, it's the ONLY option this tool actually supports for this case.

**How this was caught before it shipped:** re-ran `get_node_infos` on BOTH ends of the connection immediately after the `connect_pins` call, instead of trusting the `{"returnValue":null}` success response. The very first re-splice attempt (reconnecting the original link) then ALSO silently disconnected my new node in turn — proving conclusively (not just suspecting) that this is single-connection-per-pin behavior, not a one-off glitch. Confirmed fully fixed via a live PIE test reading real game state (`PlayerState.IndividualScoreAtRisk` climbed 0→51 on pool entry, exactly as before this change) — not just a clean compile, which would NOT have caught this class of bug (a disconnected-but-otherwise-valid exec pin is not a compile error, just silently-dead logic). **Generalizes hard: `{"returnValue":null}` / `true` from a mutating MCP call is not proof the mutation did what you intended — for anything touching an EXISTING, already-load-bearing exec chain, re-read both the source and destination pins immediately after, every time, not just on suspicion.**

New widget-authoring gotchas, all found via trial-and-error against real tool errors (never guessed silently):

1. **A widget blueprint's OWN member-variable accessor prefix is `Variables|<BlueprintName>|Get<Var>`, not `Variables|Default|Get<Var>`** — the pattern used successfully all session for every OTHER Blueprint class. `Variables|Default|GetLoudnessBar` errors "does not exist"; `Variables|WBP_HUD|GetLoudnessBar` works. Likely because widget "variables" here are actually named child-widget bindings (`ToggleWidgetAsVariable`), not plain `add_variable` members — the DSL's accessor-name resolution keys off the owning Blueprint's name specifically for this category. Check `find_node_types(graph, "Variables|")` on the ACTUAL widget graph before assuming the `Default` prefix carries over from regular-Blueprint work.
2. **UMG's plain "Text" widget (`/Script/UMG.TextBlock`) SetText/SetColorAndOpacity show up under DISAMBIGUATED type_ids you have to pick correctly among near-duplicates**: `Widget|SetText(Text)` (bare "Text" widget = TextBlock) vs `Widget|SetText` (bare, no suffix — actually `RichTextBlock`, confirmed by its self-pin reading "Rich Text Block Object Reference" when checked via `get_node_type_pins`) vs `Widget|SetText(TextBox)`/`(EditableText)`/etc. **Always verify via `get_node_type_pins` — read the `self` pin's type name — before trusting a type_id name alone**; "no suffix" is not "the plain/default one," it's whichever widget class happened to register first. Same family: `Class|Text|SetColorandOpacity` (not `Class|TextBlock|...` — the friendly class name in this node-DB is "Text", matching the palette label, not the C++ class name).
3. **Building "LABEL: value" HUD text is much easier as two separate TextBlocks (static label + dynamic value) than one concatenated string.** No `Concat_StrStr`/plain-text-`+` node was found registered under obvious names; rather than sink time into discovering Unreal's actual Text-concatenation node, split every "prefix + live number" HUD element into a static label TextBlock (set once, never touched again) plus a value-only TextBlock updated via `Utilities|Text|ToText(Integer)` straight into `SetText`. Simpler graph, zero string-building risk, and arguably better UMG practice anyway (independent Bind-ability, matches the source doc's own "many small `WBP_` widgets" philosophy).
4. **`ProgressBar.SetPercent`'s param is `Percent`, not the property name `InPercent`/`percent`** you'd expect from the property JSON schema — property names (used with `ObjectTools.set_properties`) and FUNCTION parameter names (used with `write_graph_dsl`/`create_node` keyword args) are DIFFERENT namespaces that don't always match 1:1. Same for `Widget.SetVisibility`'s enum param, which is `InVisibility` (property is `visibility`). Verify function param names via `get_node_type_pins`, don't reuse the property-schema name blind.
5. **`Class|X|SetY` type_ids returned by `find_node_types` can report a misleadingly bare `type_id` field back from `get_node_type_pins`/`get_node_infos`** (e.g. `Class|ProgressBar|SetFillColorandOpacity` node's own `.type_id` reads back as just `|SetFillColorAndOpacity`, prefix stripped) — this is cosmetic/how the tool reports self-class setters, not a sign your `create_node`/DSL call used the wrong type_id. Don't "fix" a call that already succeeded because the echoed type_id looks different from what you passed.
6. **A Blueprint can have only ONE `Event BeginPlay` node per graph** (it's a single-dispatch virtual-function override, NOT a repeatable delegate-bound event like `EventTick` felt like it might be) — `create_node("AddEvent|EventBeginPlay", ...)` on a graph that already has one **returns/relocates the EXISTING node** (confirmed via `get_node_infos`: same existing "then" connections, just moved to the new requested position) rather than erroring OR creating a second one. Always `find_nodes`+`get_node_infos` to check for an existing singleton-event node before assuming `create_node` made you a fresh one — verify by checking whether its output pins already have connections you didn't just make.
7. **To add new logic to an existing `EventBeginPlay` chain without touching/understanding its full existing content: `break_pins` the ONE link right after the event node, splice your new chain in between, then reconnect your chain's tail to whatever the event used to connect to directly.** This is the safe "insert at the very front" pattern — you don't need to trace the rest of the chain at all, and you provably can't break anything downstream since its first input is unchanged, just now sourced from your tail instead of the event directly.
8. **Widget UMG `(event EventTick ...)` DSL sugar fails** with `AddEvent|EventTick does not exist` — for `UserWidget` graphs specifically, Tick is namespaced `AddEvent|UserInterface|EventTick` (unlike a regular Actor/Component graph where bare `EventTick`/`EventBeginPlay` DSL sugar works per the blueprints skill's existing guidance). The DSL's `(event Name ...)` shorthand doesn't take a category-qualified name — fall back to raw `create_node`/`connect_pins` for any event whose `AddEvent|...` type_id has a category segment.
9. **Verified BeginPlay DOES fire reliably in PIE** — added a `PrintString` immediately after a (possibly-stale-bytecode-suspected, see the `BP_PlayerController` cold-load entries below) existing `EventBeginPlay`, and `LogsToolset.GetLogEntries` showed it firing with zero `Accessed None`/runtime-error entries across the whole PIE session afterward (confirms the new HUD-creation chain, and by extension the chain it was spliced in front of, both execute). **This is PIE evidence only, not cold-disk proof** — the project's own established rule (only a fresh process tests on-disk truth) still applies; a cold standalone relaunch is the next honest check if the "cold load" symptom resurfaces.

## 2026-07-02 (movement/hiding/swimming session) — Building `BP_BreathComponent`: five `write_graph_dsl` node-wiring gotchas, plus a real logic bug caught only by live PIE (`bIsInWater` vs `bIsSubmerged`)

### `(event EnhancedInputActionIA_X ...)` DSL syntax can only be used to relate to an EXISTING Enhanced Input event node — it cannot CREATE one
Trying to add brand-new `IA_Dive`/`IA_SwimUpDown` handlers via `write_graph_dsl`'s normal `(event EnhancedInputActionIA_Dive (...) ...)` form (which is how `read_graph_dsl` prints existing ones) fails: `AssertionError: AddEvent|EnhancedInputActionIA_Dive does not exist`. **Fix:** create the node the low-level way — `create_node` with `type_id: "Input|EnhancedActionEvents|IA_<ActionName>"` (find it via `find_node_types` filtered to `EnhancedActionEvents|IA_<Name>`), then `get_node_infos` on the new node to see its real exec pins (`Triggered`/`Started`/`Ongoing`/`Canceled`/`Completed`, plus a plain-typed `ActionValue` pin — Boolean for a Digital action, Float for Axis1D, no struct-unwrapping needed), then `connect_pins` by hand to a `CallFunction|YourHandler` node also made with `create_node`. This is the same "must delete+recreate with the correct `Input|EnhancedActionEvents|IA_X` type" rule from the original input-repair session, now confirmed to also apply to first-time creation, not just rebinding.

### `CallFunction|X` nodes for a custom Blueprint function sometimes expose an explicit "self" input pin that swallows your first positional arg
`(CallFunction|TickBreath DeltaSeconds)` (one arg, matching the function's one parameter) failed with `Could not connect pin DeltaSeconds to self` — `get_node_type_pins` showed the real pin order was `(execute, self, DeltaSeconds)`, i.e. TWO input pins beside exec, so the single positional arg bound to the wrong one. This contradicts the `Move`/`Aim` precedent from the ThirdPerson template (`(CallFunction|Move Axis_X Axis_Y)` — no self needed) — those nodes were built by the original `create_node`/`connect_pins` era of this project, not this DSL, so their "no self pin" shape isn't representative of what fresh `CallFunction` nodes for NEW functions look like. **Rule going forward: always call `get_node_type_pins` on a `CallFunction|X` node type before trusting positional args, and default to explicit keywords (`:self self :ParamName value`) for any freshly-added custom function** — cheap insurance against this exact silent-shape-mismatch class of error. Same fix applied to `Pawn|Input|AddMovementInput` (4 real input pins: self/WorldDirection/ScaleValue/bForce — the Move() function's 2-arg call was, again, pre-DSL legacy shape).

### Sequential top-level `switch` statements in one event/function body raise a false "unreachable code" error — `switch` has no single unified continuation pin the way `if` does
Adding a second, unrelated `switch` statement after an existing one in `EventOnMovementModeChanged` (checking `PrevMovementMode` right after already checking `NewMovementMode`) failed to write at all: `Unreachable code after branch/return`. A `switch` node's cases each dead-end on their own case pin — there's no single "next" pin for a following top-level statement to chain from, so the DSL compiler (correctly, if confusingly) refuses to treat anything after it as reachable. Adding an empty `(:Default)` case didn't help (this enum's switch has no Default pin — `SwitchOnEMovementMode`'s cases are exhaustive: `MOVE_None/Walking/NavWalking/Falling/Swimming/Flying/Custom`, no catch-all). **Fix: don't chain two switches — use `if (== EnumVar (Utilities|Enum|LiteralenumETypeName "EnumValue")) ... )` instead**, chaining as many `if`s in a row as needed (this form DOES support a normal single continuation, proven throughout this session in `TickBrain`/`ApplyMovementTuning`). Find the literal-maker node type via `find_node_types` filtered to the enum name (e.g. `EMovementMode` → `Utilities|Enum|LiteralenumEMovementMode`), and note the quoted-string-literal-as-enum-value trick from the DSL docs applies as the ARGUMENT to that literal node, not directly as an operand to `==`.

## 2026-07-02 (movement/hiding/swimming session) — `ObjectTools.set_properties`: to create a NEW instanced sub-object inline (e.g. an `InputModifier` on an `InputMappingContext` mapping), pass the **class path** as the `refPath` value directly — do NOT wrap it as `{"instance": "<ClassPath>"}`

### The finding
Adding `IA_SwimUpDown`'s negative-axis key mapping (`Left Ctrl` → `-1`) to `IMC_Default` needed a brand-new `InputModifierNegate` instance in the mapping's `modifiers` array (existing modifiers on other mappings are pre-existing instanced sub-objects like `IMC_Default.IMC_Default:InputModifierNegate_0`, created once, long ago, presumably in-editor). The tool's own schema doc says "For instanced sub-object properties, pass a class path as the instance member" — read literally, this means wrap it as `{"instance": "/Script/EnhancedInput.InputModifierNegate"}`. **That produced a silent no-op**: `set_properties` returned `true`, but a follow-up `get_properties` showed the modifiers array element as the literal string `"None"` — no new object was created, no error was raised either.
**What actually works:** pass the class path as the plain `refPath` value, exactly like an existing-instance reference: `{"refPath": "/Script/EnhancedInput.InputModifierNegate"}`. The tool detects the path resolves to a *class* rather than an object instance and auto-instantiates a brand-new sub-object of that class (owned by the target asset), returning it as a real new instance (confirmed: `IMC_Default.IMC_Default:InputModifierNegate_1` appeared after save, verified via `get_properties` immediately after and again via a **cold standalone process** reading `GetAll InputMappingContext DefaultKeyMappings` from disk).
**Generalizes:** whenever an MCP tool's schema description mentions a special JSON shape for "creating" something inline, don't trust the described key name at face value — verify with a `get_properties` (or equivalent) round-trip immediately after the call. A `true` return and no error do NOT mean the operation did what you intended; this is the same family of trap as the `set_pin_value`-doesn't-mark-dirty lesson below (success signals that don't mean success).

## 2026-07-02 (post-MCP-reconnect session) — CRITICAL, SILENT-DATA-LOSS RISK: `BlueprintTools.set_pin_value` does NOT mark the containing package dirty, so `save_assets` silently skips it — a pin edit can affect live runtime behavior for an entire session while never reaching disk

### The finding
Fixed a verified, high-value bug (`TickBrain`'s chase `MoveToActor` `AcceptanceRadius=120` → `55`, see the caught/detain entry below) using `set_pin_value` on 3 duplicate nodes, then `compile_blueprint` (succeeded), then `save_assets([])` (returned `true`). **The fix demonstrably took effect at runtime** — re-tested live in PIE multiple times and the Watcher's chase behavior genuinely changed (walked to ~69 units instead of stopping at 120+). ## 2026-07-02 (post-MCP-reconnect session) — CRITICAL: `write_graph_dsl` on an event ALREADY PRESENT in a shared EventGraph accumulates orphaned nodes instead of cleanly replacing them; the live exec chain is still correct (rewired to the newest nodes) but the graph balloons with dead nodes on every re-edit

### The finding
Iteratively editing `BP_SensorLight`'s `EventActorBeginOverlap` handler in-place (4 successive `write_graph_dsl` calls, each adding one more diagnostic line) grew the shared `EventGraph` from ~20 nodes to 100+. ## 2026-07-01 (same session) — Step 3 pool scoring built + wired; a real Blueprint bug caught by live PIE testing: PURE Get nodes re-evaluate at each use, they are not "read once" just because a DSL `bind` reused one node

**The bug (would have shipped silently without live testing):** `Server_BankAtRisk(PS)` computed `IndividualScoreBanked += atRisk` (correct — PlayerState side matched expectations in testing), but the parallel `GameState.TeamScoreBanked += atRisk` / `TeamScoreAtRisk -= atRisk` came out **wrong** — `TeamScoreAtRisk` stayed stuck at the pre-bank value and `TeamScoreBanked` stayed 0. `get_node_infos` proved the *graph wiring* was 100% correct — one `GetIndividualScoreAtRisk` node, its single output wired to all three consumers (`IndividualScoreBanked` add, `TeamScoreBanked` add, `TeamScoreAtRisk` subtract), exactly matching the DSL source and the "bind once, reuse" rule from `get_graph_dsl_docs`.

**Root cause:** in my original statement order, `SetIndividualScoreAtRisk(PS, 0)` (the reset) ran **before** the GameState math (which lived inside a `CastToBP_PlayerGameState` `:then` block positioned later in the function). A **pure** node (no exec pins — a plain member-variable getter) has no cached/materialized value: Blueprint re-evaluates it **fresh at each consumption point**, not once at authoring time. So `SetIndividualScoreBanked` (which executes early, before the reset) read `atRisk` correctly; but `SetTeamScoreatRisk`/`SetTeamScoreatRisk` (executing later, after the reset already zeroed the source variable) re-pulled the SAME `GetIndividualScoreAtRisk` node and got **0** instead of the frozen original value. One DSL node ≠ one evaluation — "bind once, reuse" (from the docs) only guarantees you don't author a duplicate node; it does **not** freeze a pure read against a later Set on the same property within the same function.

**Fix:** reorder so **every** consumer of a soon-to-be-reset variable's pure read happens **before** the reset statement, not just the first one. Both `Server_BankAtRisk` and `Server_LoseAtRisk` had this bug (identical shape: read `IndividualScoreAtRisk` → reset it early → later re-derive a GameState delta from the same read). Fixed by nesting the reset **last**, inside the same exec branch as the GameState update, so the reset physically cannot run before every consumer of the old value has fired.

**Generalizes to:** any function that (a) reads a mutable variable, (b) resets/overwrites that SAME variable, and (c) needs the OLD value in more than one place — the fix is always "do all reads-dependent-math first, reset last," never "bind once and trust it." This will matter again for Step 5 (AlertDirector heat/detain resets) and Step 2 finish (loudness resets) — watch for this exact shape.

**Why this was caught:** direct PIE property inspection (`ObjectTools.get_properties` on the live `PlayerState`/`GameState` instances during a running PIE session, via `set_actor_transform` teleports into the pool then the stash zone) — not just "does it compile." A clean `compile_blueprint(warnings_as_errors=true)` and a correct-looking `get_node_infos` dump both said "fine"; only reading the actual runtime numbers exposed the bug. Reinforces the project's core verification discipline (see `unreal-mcp-session` skill §3) one level deeper: even *logic*, not just *assets*, needs a live runtime check, not just a clean compile.

### `save_assets` silently returns `true` but writes nothing for corrupted stub packages
A 0-byte package that's "loaded" in the editor won't re-serialize on `save_assets` — it returns success but the file stays 0 bytes (`is_dirty` reads false). **Workaround:** `AssetTools.delete` the stub, then `DataAssetTools.create` / `BlueprintTools.create` a fresh asset at the same path. ### DSL: member variables need explicit getters; latent/multi-out nodes need keyword pins
In `write_graph_dsl`, a bare `MyVar` only resolves to locals/params — a member variable must be `(Variables|Default|GetMyVar)` or you get "Undefined variable". And positional args bind in pin order, which put a value on `self` for `Character|LaunchCharacter` (self is arg 0). Use **keyword args** (`:LaunchVelocity ... :bZOverride true`) for anything past the first data pin. Always `get_graph_dsl_docs` before first use.

### `PhysicsVolume` cannot be a content-Blueprint parent
`BlueprintTools.create` with parent `/Script/Engine.PhysicsVolume` → "Cannot create a blueprint based on the class 'PhysicsVolume'." Volumes are brush actors that must live in a level. **For a water/pool volume:** `SceneTools.add_to_scene_from_class` a `PhysicsVolume` directly into the map, scale it (actor scale resizes the brush; default is 200³ so scale 4 → 800), then `set_properties` `{"bWaterVolume": true, "priority": 1}`. `CharacterMovementComponent` switches to `MOVE_Swimming` for free when the capsule center is inside.

### The ThirdPerson template level has a raised central mound — trace before placing
Ground is **not** flat z=0 everywhere: the spawn sits on a mound (~z=210 at origin, sloping down to the z=0 outer floor), and there are scattered raised platforms (found z=200 strips). **Always `SceneTools.trace_world` (start high, end below 0) at each intended placement XY before dropping geometry**, and probe the footprint corners for a large actor. `trace_world` returns the *distance* from start to hit, so ground_z = start_z − distance.

### CaptureViewport / big describe_toolset results overflow context — extract with Python
`CaptureViewport` returns a multi-MB base64 PNG inline; `describe_toolset` for `BlueprintTools` is ~72k chars. Both exceed the tool-result limit and get spilled to a file. **Handle it:** `python3` to `json.load` the saved file, `base64.b64decode` the `image.data` to a `.png` in scratchpad, then `Read` the PNG (Read renders images). For describe dumps, extract just the tool objects you need by short name. Pass `captureTransform` explicitly (the "optional" field is required in practice) and set `annotations.gridHeight` to your ground z for readable coordinate labels.

### Material-from-scratch via MCP (translucent unlit)
`MaterialTools.create_material` → `ObjectTools.set_properties {blendMode: BLEND_Translucent, shadingModel: MSM_Unlit, twoSided: true}` → `add_expression` a `Constant3Vector` (set `constant` LinearColor) and a `Constant` (set `r`) → `connect_to_output` to `MP_EmissiveColor` / `MP_Opacity` → `recompile`. Apply per-instance via the component's `overrideMaterials` array (don't `StaticMeshTools.set_material`, which edits the shared engine mesh). Disable a decorative plane's collision via `bodyInstance.collisionEnabled = NoCollision` so the player can pass through.

### Enhanced Input mapping JSON shape (for `ObjectTools.set_properties` on an IMC)
`mappings` is an array of `{action: {refPath}, key: {keyName: "..."}, triggers: [], modifiers: [{refPath: "/Script/EnhancedInput.InputModifier<X>"}]}`. WASD from one Axis2D action: bind `D` plain, `A` with `InputModifierNegate`, `W` with `InputModifierSwizzleAxis` (default order YXZ = swap X/Y), `S` with Swizzle+Negate. Set the array in one `set_properties` call — incremental array edits fail with "insertion points are ambiguous"; `reset_properties` the array first if you need to rebuild it.
