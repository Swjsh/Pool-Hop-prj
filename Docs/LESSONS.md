## 2026-07-01 (same session) ‚Äî Step 3 pool scoring built + wired; a real Blueprint bug caught by live PIE testing: PURE Get nodes re-evaluate at each use, they are not "read once" just because a DSL `bind` reused one node

**The bug (would have shipped silently without live testing):** `Server_BankAtRisk(PS)` computed `IndividualScoreBanked += atRisk` (correct ‚Äî PlayerState side matched expectations in testing), but the parallel `GameState.TeamScoreBanked += atRisk` / `TeamScoreAtRisk -= atRisk` came out **wrong** ‚Äî `TeamScoreAtRisk` stayed stuck at the pre-bank value and `TeamScoreBanked` stayed 0. `get_node_infos` proved the *graph wiring* was 100% correct ‚Äî one `GetIndividualScoreAtRisk` node, its single output wired to all three consumers (`IndividualScoreBanked` add, `TeamScoreBanked` add, `TeamScoreAtRisk` subtract), exactly matching the DSL source and the "bind once, reuse" rule from `get_graph_dsl_docs`.

**Root cause:** in my original statement order, `SetIndividualScoreAtRisk(PS, 0)` (the reset) ran **before** the GameState math (which lived inside a `CastToBP_PlayerGameState` `:then` block positioned later in the function). A **pure** node (no exec pins ‚Äî a plain member-variable getter) has no cached/materialized value: Blueprint re-evaluates it **fresh at each consumption point**, not once at authoring time. So `SetIndividualScoreBanked` (which executes early, before the reset) read `atRisk` correctly; but `SetTeamScoreatRisk`/`SetTeamScoreatRisk` (executing later, after the reset already zeroed the source variable) re-pulled the SAME `GetIndividualScoreAtRisk` node and got **0** instead of the frozen original value. One DSL node ‚âÝ one evaluation ‚Äî "bind once, reuse" (from the docs) only guarantees you don't author a duplicate node; it does **not** freeze a pure read against a later Set on the same property within the same function.

**Fix:** reorder so **every** consumer of a soon-to-be-reset variable's pure read happens **before** the reset statement, not just the first one. Both `Server_BankAtRisk` and `Server_LoseAtRisk` had this bug (identical shape: read `IndividualScoreAtRisk` ‚Üí reset it early ‚Üí later re-derive a GameState delta from the same read). Fixed by nesting the reset **last**, inside the same exec branch as the GameState update, so the reset physically cannot run before every consumer of the old value has fired.

**Generalizes to:** any function that (a) reads a mutable variable, (b) resets/overwrites that SAME variable, and (c) needs the OLD value in more than one place ‚Äî the fix is always "do all reads-dependent-math first, reset last," never "bind once and trust it." This will matter again for Step 5 (AlertDirector heat/detain resets) and Step 2 finish (loudness resets) ‚Äî watch for this exact shape.

**Why this was caught:** direct PIE property inspection (`ObjectTools.get_properties` on the live `PlayerState`/`GameState` instances during a running PIE session, via `set_actor_transform` teleports into the pool then the stash zone) ‚Äî not just "does it compile." A clean `compile_blueprint(warnings_as_errors=true)` and a correct-looking `get_node_infos` dump both said "fine"; only reading the actual runtime numbers exposed the bug. Reinforces the project's core verification discipline (see `unreal-mcp-session` skill ¬ß3) one level deeper: even *logic*, not just *assets*, needs a live runtime check, not just a clean compile.

### Smaller findings from the same build pass
- **`SpawnActorAtTransform`/`StartPIE(startTransform=...)` silently fails if the override transform collides with existing geometry** (`LogSpawn: Warning: SpawnActor failed because of collision at the spawn location`). Don't spawn-override into a trigger volume's interior for testing ‚Äî spawn at the normal PlayerStart, then `ActorTools.set_actor_transform` to **teleport** the already-spawned pawn wherever you need it. Teleporting via `set_actor_transform` correctly re-triggers `UpdateOverlaps` (both Begin and End fired correctly across two separate teleports in testing) ‚Äî it's just the *initial spawn* that has collision-avoidance.
- **Cross-object Blueprint accessors follow `Class|BP<ClassNameNoUnderscore>|<MemberOrFunctionName>` (find via `find_node_types` from the CALLING blueprint's own graph, never assume/guess).** Same-class self-calls to a blueprint's own custom function use a *different* prefix, `CallFunction|<Name>` ‚Äî discovered when `Class|BPPoolVolume|GetCrewMultiplier` (calling itself) failed with "does not exist" but `CallFunction|GetCrewMultiplier` worked. Boolean (`bIsScoring`) member accessors strip the `b` prefix in both forms (`GetIsScoring`, not `GetbIsScoring`) ‚Äî same convention as the pre-existing `BPLoudnessComponent` accessors (`GetIsSprinting` from `bIsSprinting`). Always `find_node_types` from the actual graph you're writing into rather than pattern-matching from a different blueprint's discovered names ‚Äî the two prefixes are NOT interchangeable and guessing wastes a round trip (or worse, silently binds to the wrong node if a similarly-named type happens to exist elsewhere).
- **User-created Blueprint functions are never "pure" by default via these tools** ‚Äî even functions with no side effects (`GetEffectiveScorePerSecond`, `GetCrewMultiplier`) get exec pins (`execute`/`then`) when called, unlike auto-generated variable Get/Set accessors (which stay pure/exec-less). No tool exposes toggling the "Pure" flag. Practical effect: never inline a call to one of your own custom functions inside an arithmetic expression in DSL ‚Äî always `(bind x (CallFunction|MyFunc ...))` as its own statement first, then use `x`. Engine math/utility nodes (`Math|Float|Power`, `Array|Length`, `Array|ContainsItem`, `Networking|HasAuthority`, `Game|GetGameMode/GetGameState`, `Components|GetOwner`) ARE pure (no exec) and safe to inline.

### DSL: member variables need explicit getters; latent/multi-out nodes need keyword pins
In `write_graph_dsl`, a bare `MyVar` only resolves to locals/params ‚Äî a member variable must be `(Variables|Default|GetMyVar)` or you get "Undefined variable". And positional args bind in pin order, which put a value on `self` for `Character|LaunchCharacter` (self is arg 0). Use **keyword args** (`:LaunchVelocity ... :bZOverride true`) for anything past the first data pin. Always `get_graph_dsl_docs` before first use.

### Enhanced Input mapping JSON shape (for `ObjectTools.set_properties` on an IMC)
`mappings` is an array of `{action: {refPath}, key: {keyName: "..."}, triggers: [], modifiers: [{refPath: "/Script/EnhancedInput.InputModifier<X>"}]}`.
