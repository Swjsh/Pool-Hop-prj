## 2. High-Level System Diagram

Single-process game (no web backend). In multiplayer, one player's machine acts as a **listen server**; everyone else are clients. All truth lives server-side and replicates down.

```
[Player Input] --> [PlayerController] --request--> [GameMode (server-only, never replicated)]
                                                          |  rules: scoring, heat/escalation,
                                                          |  spawn/escape logic, threat director
                                                          v
                                              [GameState (replicated to all clients)]
                                                    team score, neighborhood alert level,
                                                    night timer, banked total
                                                          ^
                                                          |
[Threat AIControllers] --AIPerceptionComponent--> (Sight + Hearing) --> feeds GameState/AlertDirector

[PlayerCharacter] --> LoudnessComponent --ReportNoiseEvent--> AI hearing radius
      |
      +--> PoolScoringComponent (overlaps pool volume --> asks server to accrue score)
      +--> CostumeComponent (swaps mesh + applies modifier Data Asset)
```

Client → server request → server validates/owns truth → replicate back via OnRep/RepNotify. See `Docs/03_Technical_Architecture.md` §2–4 for the full model and replication gotchas.

## 3. Core Components

### 3.1. Player Character & Controller
Third-person pawn (movement via `CharacterMovementComponent` for free network prediction) + local input/cosmetic feedback only. No gameplay truth lives here.

### 3.2. Gameplay Components (reusable, data-driven)
- **LoudnessComponent** — 0–100 value; raised by tagged actions (sprint, splash, vault, sensor trip), decays over time; fires `ReportNoiseEvent` scaled by value.
- **PoolScoringComponent / PoolVolume** — trigger volume; while overlapped, requests server-side score accrual (decay + hop-streak logic resolved server-side).
- **CostumeComponent** — applies a Costume Data Asset (mesh parts + optional stat modifier).

### 3.3. AI / Threats
`AIController` + Behavior Tree + `AIPerceptionComponent` (AISense_Sight for the vision cone, AISense_Hearing for `ReportNoiseEvent`). Three alert states: Unaware → Suspicious → Alert. Detection resolution runs server-side only; clients render cones/bars from replicated state.

### 3.4. AlertDirector (on GameMode)
Aggregates loudness/detection into the neighborhood's **heat** value; decides escalation (e.g. spawns the cop threat once heat crosses a threshold).

### 3.5. Framework classes
- **GameMode** (server-only, never replicated): rules.
- **GameState** (replicated): shared truth — team score, alert level, timer, banked points.
- **PlayerState** (replicated, per player): individual score, loadout, detained status.

Technologies: Unreal Engine 5.8, Blueprints (primary) + C++ (perf hot-paths / Blueprint-inaccessible features only), UE AI Perception + Behavior Trees.

## 4. Data Stores

No traditional database. Tuning values (cone sizes, score rates, costume stat modifiers, patrol waypoints) live in **Data Tables / Data Assets** under `Content/_Project/Data/`, checked into git as content assets. No player accounts or persistent server-side storage in the MVP; any future leaderboard/progression data would live in whichever online subsystem backend we adopt (EOS or Steam) — not built yet, see §9.

## 6. Deployment & Infrastructure

- **Platform:** PC / Windows first (packaged standalone build).
- **Topology:** Listen server (one player hosts, P2P) for co-op — no dedicated server hardware, no cloud infra.
- **CI/CD:** none yet — first candidate would be an Unreal Automation Tool (UAT) build/package script once Phase 1 is stable.
- **Monitoring/Logging:** none (single-player local dev); revisit if/when online services are added.

## 7. Security Considerations

- **Server authority is the core security model** for gameplay: score, loudness, and detection are resolved server-side only; clients cannot self-report state that affects outcomes. See `Docs/03_Technical_Architecture.md` §2.
- **Unreal MCP has no authentication** — localhost is not a trust boundary. Never expose port 8000 or run the MCP server on a shared/untrusted machine.
- **`execute_tool_script` (MCP) runs arbitrary Python in-editor** with full project/disk access — treat every MCP session as privileged; commit before long sessions, review diffs after.
- **Never run Claude Code with `--dangerously-skip-permissions`** while the Unreal MCP plugin is loaded.
- No player accounts, PII, or payment data in the MVP.

## 11. Glossary / Acronyms

- **MCP** — Model Context Protocol; here, Epic's built-in UE 5.8 editor automation server (unrelated to any other MCP integration).
- **GAS** — Gameplay Ability System (UE framework; not yet in use, noted for future reference).
- **EOS** — Epic Online Services.
- **OSS** — Online Subsystem (UE's networking/session abstraction layer).
- **AI Perception** — UE5 system for AI sight/hearing senses (`AISense_Sight`, `AISense_Hearing`).
- **BT** — Behavior Tree (UE AI decision-making asset).
- **LFS** — Git Large File Storage, for binary asset version control.
- **Heat** — Pool Hop's neighborhood-wide alert/escalation value (design term, see `Docs/01_Game_Design_Document.md` §5.3).
